# RAG 신규 프로젝트 — 단계별 구현 가이드

이 문서는 **본 레포(LLM과제)와 동일한 구조**로 새 프로젝트를 처음부터 구현할 때의 순서와 각 단계에서 만들 파일·역할·주의점을 정리한 것입니다.  
PyCharm 등 새 폴더로 복사해 두고 **한 단계씩** 진행하면 됩니다.

---

## 참고하는 원본 레포 구조 (요약)

| 단계 | 새 프로젝트에 둘 파일 (권장) | 본 레포에서 참고할 파일 |
|------|-----------------------------|-------------------------|
| 환경 | `config.env`, `src/env_loader.py` | `config.env.example`, `src/env_loader.py` |
| 청킹 | `src/chunking.py` | `src/chunking.py` |
| 메타데이터(선택) | `src/metadata_filter.py` | `src/metadata_filter.py` |
| 벡터 저장 | `src/embedding.py` | `src/embedding.py` |
| RAG 답변 | `src/retriever.py` | `src/retriever.py` |
| UI | `app.py` | `app.py` |
| 고급(선택) | `src/advanced_rag.py` 등 | `advanced_app.py`, `src/advanced_rag.py` 등 |

**데이터/저장 경로 권장**

- `data/pdfs/` — 업로드·테스트용 PDF  
- `./vectordb/` — Chroma 영구 저장 (`.gitignore`에 추가)

- Python 3.12+, 프로젝트 전용 venv  
- `requirements.txt`는 본 레포와 동일하게 맞춘 상태(이미 완료했다고 가정)

## Phase 0 — 프로젝트 뼈대

**목표:** 실행과 import가 되는 빈 껍데기.

1. 폴더 생성: `src/`, `data/pdfs/`  
2. `src/__init__.py` — 빈 파일이어도 됨  
3. 루트에 `config.env` 복사(`.env`만 써도 됨). 최소 필수 키:
   - `UPSTAGE_API_KEY`  
4. (선택) `python-dotenv`를 쓰면 `.env` 로드 가능. 본 레포 `env_loader`는 **`config.env`를 직접 파싱**하는 방식.

**완료 기준:** 터미널에서 venv 활성화 후 `python -c "import langchain; import chromadb"` 오류 없음.

## Phase 1 — 환경 변수 로드 (`src/env_loader.py`)

**목표:** 앱 시작 시 `UPSTAGE_API_KEY`가 `os.environ`에 올라가 있게 하기.

**구현 요지 (본 레포와 동일 패턴)**

- `EnvironmentLoader`: `KEY=VALUE` 줄 단위 파싱, `#` 주석·빈 줄 무시  
- `load_environment(env_file="config.env")`: 파일 로드 + `UPSTAGE_API_KEY` 존재 검증  
- `get_api_key()`: 플레이스홀더(`your_upstage_api_key_here`)면 실패 처리  

**Streamlit `app.py`에서:** 키를 UI로만 받는 방식이면 이 단계는 “파일 로드는 선택”, 대신 질문/임베딩 직전에 반드시 `os.environ["UPSTAGE_API_KEY"]`가 설정되어 있어야 함.

**완료 기준:** 짧은 스크립트에서 `load_environment()` 후 `get_api_key()`가 실제 키 문자열 반환.

## Phase 2 — PDF 청킹 (`src/chunking.py`)

**목표:** PDF 경로 → **`List[Dict]`** 청크 리스트. 각 원소는 최소 다음 필드를 갖도록 통일.

```text
{
  "id": int,
  "text": str,
  "length": int,          # 선택이나 embedding 단계에서 쓰기 좋음
  "metadata": dict        # page, source 파일명 등
}
```

**구현 순서 (권장)**

1. **PyMuPDF (`fitz`)**로 전체 텍스트 추출 (본 레포: `extract_text_with_pymupdf`)  
2. **정리** (`clean_financial_text`): 조항 줄바꿈, 공백 정리 등 — 처음엔 단순 공백 정리만 해도 됨  
3. **분할** (택1 또는 병행)
   - **기본:** `RecursiveCharacterTextSplitter` — `chunk_size=1000`, `chunk_overlap=200`, 구분자에 `\n\n제` 등 금융 문서용 패턴 (본 레포 `FinancialDocumentChunker`)  
   - **확장:** `chunk_by_articles` — `제N조` 단위 분할  
4. **진입 함수:** `process_pdf(pdf_path, chunking_method="langchain"|"articles", enrich_metadata=True|False)`

**중요 — 의존성 순서**

- 본 레포의 `process_pdf`는 `enrich_metadata=True`일 때 **`metadata_filter.MetadataFilter`**를 import 함.  
- **새 프로젝트에서는 처음에 `enrich_metadata=False`로만 호출**하거나, Phase 2 완료 후 Phase 5에서 `metadata_filter`를 추가한 뒤 `True`로 전환.

**완료 기준:** 로컬 PDF 하나로 `process_pdf` 호출 시 청크 개수 ≥ 1, `text`에 한글이 의미 있게 포함.

## Phase 3 — 임베딩 + Chroma (`src/embedding.py`)

**목표:** 청크를 벡터 DB에 넣고, 유사도 검색이 되게 함.

**구현 요지 (본 레포: `FinancialEmbeddingStore`)**

- `os.environ["UPSTAGE_API_KEY"]` 필수  
- `langchain_upstage.UpstageEmbeddings` — 모델 예: `solar-embedding-1-large-passage`  
- `chromadb.PersistentClient(path=./vectordb)` + LangChain `Chroma`  
- `add_documents(chunks, metadata)`: 각 청크를 `langchain.schema.Document(page_content=chunk["text"], metadata=...)`로 변환 후 추가  
- 래퍼 메서드: `similarity_search(query, k=5)`, `get_collection_info()`, **`load_existing_vectorstore()`** — 컬렉션 존재하고 `count > 0`이면 True  

**완료 기준:** 인덱싱 후 `similarity_search("이자율", k=3)` 등으로 `Document`가 반환됨.

---

## Phase 4 — 검색 + LLM 답변 (`src/retriever.py`)

**목표:** 질문 → (선택) 쿼리 보강 → 검색 → **컨텍스트 기반 답변** + 출처.

**구현 요지 (본 레포: `FinancialDocumentRetriever`)**

- `langchain_upstage.ChatUpstage` — 본 레포는 `model="solar-pro2"`, `temperature=0.1`, `max_tokens=4000`  
- `RetrievalQA.from_chain_type` — `chain_type="stuff"`, `retriever=vectorstore.as_retriever(search_kwargs={"k": 5})`, `return_source_documents=True`  
- 프롬프트: “문맥에만 근거”, “없으면 약관에서 찾을 수 없다” 등 (본 레포 `prompt_template` 참고)  
- (선택) `enhance_query_with_finance_terms` — 동의어·카테고리별 용어 덧붙이기  
- **공개 API:** `answer_question(question) -> dict` — 최소 키: `answer`, `sources` (리스트), `question`  

**완료 기준:** 인덱싱된 DB에 대해 질문 시 자연어 답변 + `sources`에 청크 요약/메타데이터.

---

## Phase 5 — 메타데이터 보강 (선택, 본 레포와 동일 동작)

**목표:** `chunking.process_pdf(..., enrich_metadata=True)`가 깨지지 않게 함.

- 본 레포 `src/metadata_filter.py`의 `MetadataFilter.enrich_chunk_metadata` 패턴 참고  
- 청크별 `metadata`에 `source`, `document_type`, 필터용 필드 등 추가  

**완료 기준:** `enrich_metadata=True`로 처리한 뒤에도 Chroma `add_documents`가 성공하고, (선택) 메타데이터 필터 검색이 가능.

---

## Phase 6 — Streamlit UI (`app.py`)

**목표:** 사용자가 키 입력 → PDF 업로드·벡터화 → 질문 → 답변·출처 확인.

**본 레포 `app.py` 흐름 참고**

- `st.session_state`: `vectorstore`, `retriever`, `chat_history`, `api_key_set`, `auto_submit`  
- 사이드바: API 키 → 탭으로 PDF 업로드 / 기존 vectordb 로드  
- `st.form`으로 질문 입력(Enter 제출)  
- 예시 질문 버튼 → `auto_submit` 플래그로 즉시 실행 패턴  

**완료 기준:** 브라우저에서 한 사이클(업로드→처리→질문)이 끝까지 동작.
